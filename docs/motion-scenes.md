# Webine motion and scene configuration

The current homepage uses one coordinated scene controller. Settings and layout compositions live in `src/config/experience.ts`.

| Feature | Current state | Dedicated stage |
|---|---|---|
| General scroll reveals | GSAP controller for public content, with section-owned timelines taking precedence | Public routes |
| Particle narrative | Persistent GPU layer on tablet and desktop, section-owned 2D canvases on phones | Homepage narrative |
| Smooth scrolling | One Lenis runtime with weighted wheel interpolation, extreme-input normalisation and unchanged synchronised touch values | Stage 3 onward |
| Signal Grid response | Enabled only for the hero and closing CTA | Homepage narrative |
| Page loading and transitions | Asset-aware Webine wordmark convergence followed by a clip wipe | Public routes and preview; Admin deliberately uses functional inline states only |

The static HTML and CSS composition must remain complete when every feature is disabled. Future implementations should read this configuration instead of creating separate hardcoded switches inside page components.

Public content outside a section-owned timeline uses `GsapRevealController`. Elements opt in with `data-gsap-reveal="copy"`, `card` or `media`, may provide a bounded `data-gsap-delay` and may opt into scrubbed media parallax. The controller does not mount until the page loader has fully wiped away, then creates reversible ScrollTrigger entries and observes asynchronously loaded CMS cards. A coalesced refresh runs after CMS insertion so late Project cards receive correct viewport measurements. Elements already in the opening viewport use only a 0.08-second sequencing cushion because the loader now owns route entry. Entry reveals animate opacity rather than `autoAlpha`, so offscreen links and controls remain available to keyboard and accessibility APIs. A focus-within override immediately exposes any focused reveal target. A composition with its own coordinated timeline declares `data-gsap-managed="true"`, which prevents a second reveal system from animating the same descendants. Particle renderers never receive these attributes.

Every route mounts one `PageLoadProvider`. Its visual language uses the Webine name rather than an unrelated spinner. `WEB` and `INE` use the same primary typeface and begin apart. Real readiness progress is expressed only by the whitespace closing between them until they read as one `WEBINE` wordmark. The composition stays small and minimal on one flat slate background, with no percentage, line, dots, bar, glow, grid or extra status copy. The loader has a short minimum presence so it reads as choreography, but its progress comes from real completion phases rather than a fixed five-second counter. A 12-second ceiling releases a degraded but usable route if a dependency fails. Body scrolling remains locked only while the loader is mounted. The final left-to-right clip wipe restores scrolling and then marks the page ready.

The Home `HeroEntranceTimeline` reads that ready state. While the loader is visible it holds particle intro progress at zero and marks the hero `waiting`. Only after the wipe finishes does the breathing pause begin, followed by the particle gather and semantic interface reveal. This preserves the user-approved breathing moment instead of spending it invisibly behind the loading screen.

Project imagery uses one axis-aware image-parallax contract on desktop, tablet and mobile. Works cards and case studies select vertical travel on an inner media wrapper, moving from minus eight to plus eight percent on wider layouts and minus six to plus six percent on phones. The wrapper has eight percent image bleed so this movement cannot expose an edge. The horizontally pinned Home runway selects horizontal travel and derives a small image offset from each card's distance to the viewport centre. Shared tweens use clamped viewport ranges, refresh invalidation and `immediateRender: false`, preventing a pinned scene above an image from producing an early transform or direct-load jump. Separate wrappers keep hover scaling and scroll translation from competing for the same transform.

The same controller supports restrained scrub modes beyond media, including editorial drift, a slow floating-card layer and an orbit layer for compositions that need them. Works uses drift only on its outlined background word. Project-card text no longer drifts because it made the information harder to read. Contact moves the complete form from minus 72 to plus 96 pixels on wider layouts and minus 24 to plus 36 pixels on phones. The form owns pixel translation only, because combining it with height-based percentage travel made the two directions cancel on long mobile forms. CSS never transitions the form's transform because GSAP owns that property. Reveal and parallax transforms never share the same element.

Every formed particle object has a named control surface in `src/config/experience.ts`. Home uses `heroModel`, `reachModel`, `interludeObject` and `closingModel`, About uses `aboutHead` and `aboutPortrait`, and Services uses `servicesOrb`. Responsive Home anchor positions and scene scale remain in `particleSceneConfig` in the same file. Section 4's `interludeObject` controls its elliptical band count, x and y radii, band spacing, tube thickness and rotation. Desktop generates the procedural target from these values, while mobile applies the same authored rotation to its lightweight baked target.

GSAP, ScrollTrigger, Lenis updates and the Home-owned timelines use the shared `scroll-runtime` registration. Webine keeps one complete motion system and does not branch any route on `prefers-reduced-motion`. Responsive differences use input capability, viewport geometry and renderer ownership without removing the intended story. Essential content remains semantic HTML outside decorative canvases.

## About page scenes

The About hero uses a section-owned sticky Three.js scene rather than joining the Home narrative canvas. A 9,000-point binary target is derived from the supplied `simple_head.glb` at build-preparation time. Desktop keeps all 9,000 points, while phone runtime geometry selects 5,600 evenly distributed points and caps DPR at 1.05. The source geometry is recentered around its particle centroid before rotation. ScrollTrigger is created in the layout phase and refreshed before its progress is assigned, so the first visible pose and every later return to the top use the same centre. Scroll rotation derives from current section progress rather than accumulating wheel deltas, while fine-pointer movement adds damped whole-object tilt and a local shader bulge. Every point also receives a seeded spatial offset plus a differently phased, multi-frequency three-axis path, so the face has a porous edge and visible local life while retaining its designed orientation. Scroll progress disperses the form into a spherical field. The model target is about 105 KB, so visitors do not download the source GLB. The visual wrapper fills the complete visible hero frame, allowing the dispersed field to travel across the viewport instead of being clipped inside a centred model box. The canvas is decorative, lazy, paused offscreen and isolated from semantic page content.

Kidson and Whai Hoe each use one aspect-ratio frame containing one shared moving media wrapper. That wrapper owns the real full-colour HTML image at the base, an SVG-rendered grayscale copy above it and the independent 2D particle canvas at the top. A pixel-aligned person mask is sampled only where inside and outside pixels meet, producing a contour rather than a filled subject. Stable seeded particles begin across a wide field below and beside the frame, then rise individually with staggered two-frequency curl into the silhouette, matching the cyan-to-blue stipple of the main Webine particle language without mounting another Three.js scene. Desktop is capped at 900 contour points and the visible portrait frame is capped at 25rem. Phones retain a 360-pixel mask sample, a five-pixel contour step and a hard 595-point cap. Mobile draws at 30 FPS, 1× DPR and skips the secondary glow pass. Every point receives its own seeded float speed, horizontal and vertical amplitude, curl strength, direction, phase and flow offset. Travel uses stronger local currents, wider lateral freedom and varied curling paths; settled points retain smaller independent drift plus subtle size and opacity breathing. The renderer continues updating through the configurable completed-outline hold and particle fade, so the silhouette never freezes before the photograph appears. Fine-pointer movement erases the SVG grayscale mask through displaced radial waves layered with a decaying trail. Each wave expands from the pointer with a soft irregular boundary, producing a liquid colour bloom and residual wake instead of a fixed circular cutout. Native pointer listeners are attached directly to the portrait frame. Mouse and pen can reveal colour, while `pointerType="touch"` cannot. Touch keeps the complete grayscale portrait without a simulated hover state or toggle. Outline, hold, fade, image reveal and hover-wave timings are centralised under `particles.aboutPortrait` in `src/config/experience.ts`. A refresh-safe vertical parallax moves the shared media wrapper from -3.8% to 3.8% on desktop and -2.2% to 2.2% on phone. The wrapper is scaled to 1.08 inside the clipped frame, preventing exposed edges. Because the images, SVG mask and particle canvas are children of that one transform owner, the silhouette cannot drift away from the portrait during scroll.

## Services page scenes

Services uses one section-owned `ServicesChapterController`. At desktop width, a sticky rail identifies five real offer chapters: web design and development, website redesign, monthly maintenance, SEO foundations and branding support. Normal vertical scrolling moves through the full-height copy. Each chapter scrubs its hierarchy into place and changes the active rail state at viewport centre. The former outlined circle is now a 780-point cyan-to-blue orb. Its chapter rotation stays between -0.3 and -1.1 radians. The endpoint is equivalent to the earlier authored facing without replaying the former full turn, while a bounded idle reversal and fine-pointer tilt keep it responsive. Nearby points bulge around the pointer. Each point owns a small seeded spatial offset and a differently phased multi-frequency three-axis orbit, creating a wider porous ring without moving the field as one unit. On mobile the rail is removed, chapters keep normal document flow and the content hierarchy remains complete. The Services hero receives a separate 84-point ambient canvas behind its semantic content. The page reuses the CMS-backed four-step process below this scene rather than duplicating process copy.

## Public cursor scene

Public routes mount one `KineticCursor` through `SiteShell`; Admin never mounts it. At desktop widths with a true fine pointer, a 10-pixel higher-opacity slate core follows pointer input with tight damping while a 44-pixel lower-opacity halo follows more loosely. Entering a real link, button or interactive control morphs the halo toward a clamped version of that control's geometry and contracts the core. Pressing strengthens the response. The layer is decorative, `pointer-events: none`, hidden before the first pointer event and absent below 48rem, on touch or on coarse-pointer devices. Native focus, hit targets and keyboard feedback remain unchanged.

`SiteShell` resolves its real shell element through a callback ref before mounting `GsapRevealController`. The controller receives that element directly rather than reading a parent ref during its own first layout effect. This is a production requirement, not an implementation preference: a child layout effect can run before the parent host ref is available, while the development lifecycle may run the effect again and conceal the defect. A production build is only motion-ready when `.site-shell` exposes `data-gsap-controller="ready"` and the expected reveal or parallax elements expose `data-gsap-motion-ready`.

`AmbientParticleField` is one lightweight deterministic 2D canvas rather than many animated DOM nodes. Home uses 84 points, Contact uses 76, Services uses 84 and the Works galaxy uses 138. Mobile keeps the complete requested density. Each point follows two seeded frequencies across a wider x and y radius, producing slow electron-like loops instead of a short shared oscillation. The shared CPU surface sampler moves coherent deep-blue, cyan and light-blue regions across the field. The renderer caps DPR at 1.25, limits updates to 30 FPS below 600 px and 45 FPS otherwise, pauses outside the viewport or in a hidden document and avoids React state in its frame loop. Fine-pointer movement adds a damped field parallax. Counts, travel radii, DPR and frame ceilings are read from `particles.ambientField`, while field timing is read from `particles.surfaceField` in `src/config/experience.ts`. It never receives GSAP attributes or loads Three.js. The Home field remains at ambient layer 0, the Webine logo renderer at layer 1 and hero content at layer 2.

Works wraps its index and valid case-study states in one `GalaxyBackdrop`. The backdrop is fixed to the viewport beneath transparent section surfaces, so its 138-point star field and broad cyan or blue nebula remain spatially stable while the portfolio scrolls. The nebula must read as a visible atmospheric horizon rising through the lower half of the viewport, not as an almost imperceptible tint. Phones keep the same point count and full nebula opacity. The former commissioning panel has been removed, so the atmospheric field continues directly from the project grid towards the footer. This adapts the attached Lenis showcase reference rather than copying its WebGL implementation. Webine uses the shared efficient ambient canvas to avoid loading a second Three.js experience. The implementation research is recorded in `research/2026-07-16-lenis-showcase-teardown.md`.

Directional interface marks use `DirectionalArrow`, including oversized decorative arrows. Do not place Unicode arrows in React copy or pseudo-element content because iOS and other platforms may substitute an emoji-style glyph. The source-wide regression check rejects Unicode arrow characters in interface files.

Particle colour remains outside GSAP. GPU forms derive two broad moving bands from their live object-space position and retain a small seeded residual term, so deep blue, cyan and light blue remain visible together. A separate slower field controls coherent density pockets without deleting points. Phone and CPU canvas objects sample the same field contract during active frames. Light phone scenes use darker blue endpoints and a higher density floor so the silhouette remains legible. Closing formation begins at 1.28 viewport heights so the planet is already gathering before its anchor enters the visible frame.

## Current scene contract

The Home route registers `hero`, `reach`, `work`, `interlude`, `process` and `closing`. A single GSAP timeline owns the first-load breathing period, hero gathering and interface reveal. It runs once per browsing session, uses a shorter return entry and completes immediately when the visitor scrolls or interacts early. After the hero entrance, each visual-form section owns its point, formation interval, hold and dispersion interval.

The hero-to-reach cover uses two implementations behind one visual contract. Desktop and fine-pointer devices use a ScrollTrigger pin with no added scroll spacing. Phones and coarse-touch devices use native CSS sticky inside the bounded `.hero-reach-cover` wrapper. Both keep the hero composition held while the rounded Section 2 surface rises over it. This split avoids relying on a scripted pin during real iPhone touch scrolling, where browser emulation does not reproduce all WebKit viewport and compositor behaviour. The explicit layer order is hero background and Signal Grid, particles, then hero typography and controls.

As Section 3 approaches, an inner entrance layer begins up to 14 viewport-height units above its resting composition, capped at 120 px, and settles downward by the time the section top reaches 30 percent of the viewport. The pinned stage itself is never translated, preventing the entrance parallax from competing with runway pinning or chapter expansion transforms.

The selected-work runway pins on desktop, tablet and mobile. Normal vertical scrolling moves the rendered track horizontally, with no custom wheel interception or nested touch scroller. The final chapter card stops at the viewport centre after 70 percent of the scrubbed timeline. Particles fade fully during the horizontal sequence, return as a reduced-density field during expansion and begin gathering before the pin releases.

Its entry timeline reveals the header first, then each published Project card at 0.28-second offsets with its media clip following 0.04 seconds later. Chapter 04 enters only after the final Project card. The runway owns this sequence and is excluded from the global reveal controller.

Runway entry also uses opacity without hiding visibility. Focusing a Project card reveals it immediately. Cards become `inert` only after chapter 04 begins expanding and reverse scrolling removes `inert` when the horizontal sequence returns.

The HTML process line and node states use their centre-based progress. Particle motion uses two additional geometry-derived values, an earlier intake approach and the centred outlet release. All three are recalculated from the same live line rectangle, so reverse scrolling reverses the illusion without callback-owned state.

The timeline line remains below its cards and circular nodes at every breakpoint. Each node is outside the card-content reveal, so all four circles remain present on the bar even before their copy animates in. Active-state measurement reads the circle itself, not the card top. Desktop odd and even offsets are calibrated independently so both left and right cards centre on the one-pixel line. This also prevents the fill from painting over a mobile node and activates each stage when its actual circle reaches viewport centre.

The refined particle timing uses two geometric measurements from the same timeline line:

- CSS `--timeline-progress` begins when the line top reaches viewport centre and controls fill and nodes.
- `timelineIntakeProgress` begins earlier and controls release from the interlude, dispersed travel, point gathering and staggered contact fade.
- `timelineReleaseProgress` begins when the line bottom reaches viewport centre and controls only point-origin emission and dispersion. Closing formation belongs to the closing section point.

Hero, reach, interlude and closing use independent motion ranges stored beside their own responsive anchor configuration. When a point enters the viewport, that section's particles begin forming around it. Formation reaches one when the point reaches viewport centre. The object then stays attached to the moving point until it crosses the top edge, where that section's dispersion begins. Reverse scrolling derives the same states from the same point geometry.

The selected-work chapter remains an intentional shape exception. It owns no settled target. Visibility starts falling once reach dispersion reaches 18 percent and reaches zero by 78 percent, before the runway pin begins. It remains invisible through horizontal work, then fades back from 70 to 84 percent while the chapter card expands.

The final runway item is the preview for chapter 04 rather than a generic end card. Once the card reaches the viewport centre, horizontal movement stops. Continued vertical scroll enlarges the card shell, fades its compact preview and aligns it exactly over the real interlude before the pin releases. The card does not render a second copy of the full chapter. Other project cards fade during expansion, so the one real interlude layout takes over without doubled content or a dark card overlap.

When runway progress reaches the completed `expanded` phase, the generated pin spacer returns to the normal stacking level. This is required because GSAP copies the runway's earlier scene layer onto the spacer. Without that release, the now-empty expanded card surface can remain above the real interlude and hide its text.

The expanding card also owns a reversible formation value from 70 to 88 percent of runway progress. It forms the interlude orbit at the real section point while that point is still below the viewport. The point therefore enters already formed, with no virtual position or second particle target.

Particle uniforms use a restrained damping value and the runway uses a 1.45 second scrub. Lenis uses `lerp: 0.075` and a `0.92` wheel multiplier. Wheel deltas pass through an 84-pixel hyperbolic cap, which keeps small trackpad input responsive while preventing one extreme event from skipping a complete scene. This is input normalisation inside Lenis, not another physics loop, timer or scroll lock. Touch events bypass the cap and retain `syncTouchLerp: 0.075`, `touchInertiaExponent: 1.7`, `touchMultiplier: 1` and overscroll. Mobile particles are section-owned 2D canvases rather than the persistent WebGL layer. They contain 2,200 independently sampled points per target, redraw for narrative changes and nearby ambient motion, and use a 90 ms measurement-settle window.

Zero-valued formation and dispersion uniforms release shader ownership immediately. Values still ease inside an active range, but a later scene cannot remain active through an asymptotic value after reverse scrolling.

The interlude disperses when its own point crosses the viewport top. `timelineIntakeProgress` then follows the live top point of the HTML line. Each particle has a stable contact threshold and fades only on contact, with most particles absorbed before the line top reaches viewport centre. Outlet release begins when the line bottom reaches viewport centre. Particles fade in from that point and disperse. When the closing point enters, its independent formation progress gathers that field into the colony planet mesh target. Formation runs from viewport Y `1.2` to `0.62` on desktop and tablet, while mobile uses the earlier `1.2` to `0.3` window.

The complete object slowly self-rotates while held. Fine-pointer movement adds restrained object travel and tilt, while the shader adds a local depth bulge. These idle offsets are local to the section anchor rather than a replacement for it. The closing object never fades, so the fully formed planet scrolls away with the section instead of following the viewport into the footer.

Future tablet and desktop refinements must keep the scene anchors and shared controller boundary. Phone refinements must keep the section-owned canvas model unless real-device evidence supports a different approach. New visual states should extend the same progress store rather than introduce an unrelated scroll controller.
